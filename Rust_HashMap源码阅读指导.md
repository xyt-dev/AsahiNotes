## 第一章 导论

在正式扎入茫茫的源码海洋之前，我们需要先搞清楚两件事：Rust 为什么要用现在的架构？以及，我们到底该去哪里看源码？

### 1.1 告别链表：为什么我们需要 SwissTable？

如果你看过早期 Java（JDK 1.8 之前）或者 C++ STL 的源码，你会发现传统的 HashMap 几乎都采用**“数组 + 链表（拉链法）”**来解决哈希冲突。

这种设计的逻辑非常直观，但在现代计算机体系结构下，却暴露出了一个致命的性能缺陷：**对 CPU 缓存（CPU Cache）极其不友好。**

现代 CPU 读取内存并不是一个字节一个字节读的，而是按缓存行（Cache Line，通常为 64 Bytes）成块加载。拉链法中的链表节点在堆内存中是离散分配的。当 CPU 顺着指针遍历链表时，下个节点的物理地址往往远在天边，导致极高的 Cache Miss（缓存未命中）率。此时，运算速度极快的 CPU 只能停下来，苦苦等待缓慢的主存（RAM）把数据搬运过来，这就是所谓的“内存墙”。

为了打破这堵墙，Google 的 C++ 核心库 Abseil 团队提出了一种极其暴力的现代哈希表架构——**SwissTable**。

SwissTable 彻底抛弃了链表，它的核心哲学只有八个字：**内存紧凑，暴力并行。**

1. **开放寻址（Open Addressing）：** 所有的数据（Key 和 Value）都极其紧凑地存放在一整块连续的数组内存中，最大化利用 CPU 缓存行。
    
2. **元数据与 SIMD 降维打击：** 它抽取了每个 Key 哈希值的前 7 个 bit 作为“指纹”，存放在一个连续的控制组（Control Bytes）中。查找时，利用现代 CPU 的 SIMD（单指令流多数据流）指令集，**1 个 CPU 周期就能同时对比 16 个指纹**。
    

Rust 的核心开发者们敏锐地察觉到了这种算法的威力，于是将其完全移植到了 Rust 中，这就是如今标准库 `HashMap` 的底层基石。

### 1.2 看破外壳：Rust 标准库的“金蝉脱壳”

带着对 SwissTable 的初步认知，我们打开 Rust 标准库的源码文件 `std::collections::HashMap`。

你会惊讶地发现，标准库里并没有密密麻麻的算法逻辑，它的核心定义异常干净，甚至可以说是“简陋”：

```Rust
// 如果当前不是测试环境，则向编译器注册此结构体为特殊的诊断项 "HashMap"
#[cfg_attr(not(test), rustc_diagnostic_item = "HashMap")]
// 标记此结构体为稳定版 API，从 Rust 1.0.0 版本开始就存在了
#[stable(feature = "rust1", since = "1.0.0")]
// 告诉编译器：这个类型的析构函数（Drop）是“微不足道”的（仅释放内存，无其他副作用）
#[rustc_insignificant_dtor]
pub struct HashMap<
    K,
    V,
    S = RandomState,
    // 标记泛型 A (自定义内存分配器) 是不稳定的实验性功能，追踪 Issue 为 #32838
    #[unstable(feature = "allocator_api", issue = "32838")] A: Allocator = Global,
> {
    base: base::HashMap<K, V, S, A>,
}
```

这里定义了哈希表的四大护法（泛型参数）：

- **`K` 和 `V`：** 键与值。Rust 编译器在文档中严格契约：`K` 必须实现 `Eq` 和 `Hash` trait，保证可比较性和哈希一致性。
    
- **`S = RandomState`：** 哈希状态机。默认采用 SipHash 算法，并在此基础上注入了强随机种子，这是 Rust 为了防范恶意构造冲突（HashDoS 攻击）而设置的安全底线。
    
- **`A = Global`：** 内存分配器。这赋予了极客玩家在特殊场景下（如嵌入式或高性能网关）接管哈希表内存分配的权利。
    

结构体内部，唯独只有一个 `base` 字段。顺着源码头部的 `use` 声明，一切真相大白：

Rust

```
use hashbrown::hash_map as base;
```

**这就是 Rust 标志性的“零成本抽象（Zero-cost Abstraction）”。**

Rust 官方并没有在标准库里闭门造车，而是直接引入了开源社区经过极其严苛测试的 crate —— **`hashbrown`**。标准库的 `HashMap` 仅仅是一层极薄的 Wrapper（包装器），它的内存布局与底层的 `hashbrown::HashMap` 100% 一致，且所有的调用在编译期都会被 `#[inline]` 内联展开，不存在任何额外的函数调用栈开销。

### 1.3 源码寻址指南

基于上述发现，我们的源码阅读路线图就非常清晰了：

- **表层接口API：** 看 `std::collections::HashMap`（理解 Rust 是如何向开发者暴露安全接口的）。
    
- **底层物理引擎：** 必须去阅读 `hashbrown` 库（特别是里面的 `raw::RawTable`），这才是哈希表真正管理内存、处理冲突、执行 SIMD 并行查找的战场。

## 第二章 核心结构拆解：`RawTable` 与内存魔法

当我们翻开 `hashbrown::raw` 模块，我们会发现哈希表的一切状态，都浓缩在两个极其精炼的结构体中：`RawTable` 和 `RawTableInner`。

### 2.1 大道至简：代码膨胀的“绝缘体”

在泛型编程中，有一个臭名昭著的问题叫**单态化代码膨胀（Monomorphization Bloat）**。如果把所有控制逻辑都写在带有泛型 `<T>` 的结构体里，编译器会为每一种类型（比如 `HashMap<i32, i32>` 和 `HashMap<String, String>`）复制并编译一份完全相同的底层扩容、寻址代码，导致编译产物体积暴增。

`hashbrown` 为了做到极致的优化，巧妙地将结构体拆成了两层：

**第一层：带泛型的外壳 `RawTable<T>`**

Rust

```
pub struct RawTable<T, A: Allocator = Global> {
    table: RawTableInner,     // 真正的状态机，去掉了泛型 T！
    alloc: A,                 // 内存分配器
    marker: PhantomData<T>,   // 幽灵数据，骗过编译器，假装我们拥有 T 以保证 Drop 安全
}
```

`RawTable<T>` 只负责和具体类型 `T`（也就是键值对的元组 `(K, V)`）相关的操作，比如内存分配大小计算、读写具体类型的数据。

**第二层：抹除泛型的无情内存机器 `RawTableInner`**

Rust

```
struct RawTableInner {
    bucket_mask: usize,
    ctrl: NonNull<u8>,
    growth_left: usize,
    items: usize,
}
```

这才是哈希表运转的“心脏”。它完全不知道泛型 `T` 的存在，只和底层的字节（Bytes）打交道。我们来逐一拆解这 4 个字段：

- `items`：当前哈希表中实际存在的元素个数（相当于 `len()`）。
    
- `growth_left`：在触发下一次扩容前，还能插入多少个元素。它和容量（Capacity）有关，但也受装载因子（Load Factor）和墓碑（Tombstone）机制的影响。
    
- `bucket_mask`：**容量掩码**。在 SwissTable 中，容量强制为 2 的幂次方。如果容量是 16，掩码就是 `16 - 1 = 15`（二进制 `0000_1111`）。任何哈希值只需 `hash & bucket_mask` 就能瞬间完成取模运算，避免了极其昂贵的 CPU 除法指令。
    
- **`ctrl`：指向控制字节（Control Bytes）的绝对裸指针。这是整个结构体里唯一的一个内存指针！**
    

等等……发现盲点了吗？ **存储实际键值对的 `data` 指针去哪了？！**

### 2.2 内存黑魔法：拼接 Layout 与指针倒推

这正是上一版草稿中让你觉得惊艳的底层魔法。在高级语言里，我们理所当然地认为应该有一个数组存控制位，一个数组存数据。但在 `hashbrown` 中，它们被**强行糅合在了一块连续的物理内存中**。

当哈希表初始化或扩容时，它会执行类似以下的堆内存申请逻辑（基于 `calculate_layout`）：

1. 计算数据槽 `[T]` 需要的内存大小。
    
2. 计算控制组 `[u8]` 需要的内存大小（注意：这里会额外分配 16 个字节的**镜像缓冲区 Mirror Bytes**，以防止 SIMD 一次性读取 16 字节时在数组尾部发生越界）。
    
3. 通过操作系统的对齐算法，把控制组紧紧贴在数据槽的后面，向系统申请一整块内存。
    

**内存布局全景图（ASCII 视角）：**

Plaintext

```
| <-------------------------- 一整块连续的堆内存 ----------------------------> |
| [ 数据槽 Data Slots 区 (T) ]                 [ 控制组 Control Bytes 区 (u8) ]
+-------+-------+-------+--    --+-------+-------+-------+-------+----++
| T[0]  | T[1]  | T[2]  |  ...   | T[15] | C[0]  | C[1]  | ...   | C[15]|| C[16..31] (镜像)
+-------+-------+-------+--    --+-------+-------+-------+-------+----++
^                                        ^
|                                        |
|--- data 的逻辑位置                       |--- ctrl 指针的真实指向 (RawTableInner 唯一保存的指针)
```

因为 `RawTableInner` 里只保存了 `ctrl` 指针，当它需要读取真实数据时，它会使用**指针倒推（Pointer Arithmetic）**。 由于 `ctrl` 总是位于数据的尾部，系统只需将 `ctrl` 指针向低地址方向回退 `(bucket_mask + 1) * sizeof(T)` 的距离，就能在 O(1) 的时间内计算出数据槽的起始指针 `data`。

**为什么要这么做？**

1. **省内存：** `RawTableInner` 省去了一个指针（在 64 位系统上省了 8 字节），让结构体更紧凑，更容易塞进 CPU 寄存器。
    
2. **缓存命中：** 控制字节和数据在同一块内存分页（Page）中。当你查找到控制字节时，紧接着去拿数据，这块内存极大概率已经被硬件预取（Prefetch）到了 L1/L2 Cache 中。
    

### 2.3 状态机密码：什么是 Control Bytes？

既然 `ctrl` 指针如此重要，那它指向的这些 `u8` 字节到底存了什么？ 每一个槽位（Bucket）都对应一个 1 字节（8 bits）的控制位。这 8 个比特位就像是槽位的“红绿灯”兼“指纹库”。

在源码中，它定义了 4 种魔法状态（通过最高位，即 MSB 来区分）：

- `EMPTY` (`0b1111_1111` 或 `0xFF`)：**空槽位**。最高位是 1。表示这个位置从来没放过数据。
    
- `DELETED` (`0b1000_0000` 或 `0x80`)：**墓碑（Tombstone）**。最高位是 1。表示这里曾经有数据，但被删除了。它是解决二次探查（Quadratic Probing）链条断裂的关键。
    
- `FULL` (`0b0xxx_xxxx`)：**已占用**。**最高位是 0**。当最高位是 0 时，剩下的低 7 位（`xxx_xxxx`）存储的是**该节点 Key 的哈希值的高 7 位特征（H2）**。
    

当你插入一个 Key 为 `"Rust"` 的数据时，系统算出了一个 64 位的哈希值。它会把这 64 位劈开：

- **高 57 位（H1）：** 用于在数组中定位起始的 `bucket_index`。
    
- **低 7 位（H2）：** 作为“短指纹”，写入到 `ctrl` 字节组的 `0b0xxx_xxxx` 中。
