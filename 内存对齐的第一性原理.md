# 对齐的本质：从字段同余约束到复合类型布局

内存对齐常被解释为“为了让 CPU 访问更快”或“为了避免跨 cache line”。这些说法并非错误，但它们停留在结果层，而不是本质层。

所以为何结构体中的字段要满足各自的对齐要求？如果一个对象内部包含多个字段，而每个字段都有自己的对齐要求，那么整体对象的起始地址应该满足什么约束，才能保证所有字段在运行时都自然对齐？

> **自然对齐**通常指一个基本类型的对齐要求等于它自身大小。  

本文用小学2年级都能听懂的语言从第一性原理出发彻底讲解清楚。

---

## 1. 什么叫对齐要求(Alignment)

所谓一个类型 `T` 的对齐要求为 `A`，本质上是说：

$$内存对象的首地址 base\_addr 必须满足： base\_addr ≡ 0 (mod A)$$
例如：

> u8  的对齐要求通常是 1 字节
> u16 的对齐要求通常是 2 字节
> u32 的对齐要求通常是 4 字节 
> u64 的对齐要求通常是 8 字节

满足对齐要求的作用主要概括为几个方面：

* 类型系统承诺：T 对象会放在 A 的倍数地址上
* 编译器利用：它可以假设 T* 指针满足 A 对齐
* 硬件受益：大多数访问不会落入代价很高的未对齐路径
* 编程受益：跨平台、可优化、可静态推理的布局规则

所以，对齐其实是软件和硬件之间的契约。

## 2. 单个字段的对齐：局部同余约束

设某个字段类型为 F，它的**对齐要求为 K**。

那么这个字段在内存中的实际地址 addr(F) 必须满足：

$$addr(F) ≡ 0 (mod ~K)$$

如果这个字段位于某个复合对象内部，那么它的地址可以写成：

$$addr(F) = base + offset(F)$$
其中：

$$base = 整个复合对象的起始地址$$
$$offset(F) = 字段 F 相对于对象起始地址的偏移量$$

因此字段的对齐条件变成：

$$base + offset(F) ≡ 0 (mod ~K)$$

由于编译器在布局结构体时，能直接控制的是 $offset(F)$，不能直接控制每个实例运行时的 `base`。

所以编译器只会在类型布局内部安排：

$$offset(F) ≡ 0 (mod ~K)$$

这样一来：

$$addr(F) = base + offset(F) ≡ base (mod ~K)$$

要让字段最终地址满足：

$$base + offset(F) ≡ 0 (mod ~K)$$

就只需要保证：

$$base ≡ 0 (mod ~K)$$

而复合类型对象整体的 `base` 一般由内存分配器(allocator)、栈帧布局或全局变量布局保证。

## 3. 把多个字段约束压缩成一个 base 约束

一个结构体可能包含多个字段：

```rust
struct S {
    a: u8,  
    b: u32,  
    c: u64,  
}
```

假设它们的对齐分别是：

align(u8)  = 1  
align(u32) = 4  
align(u64) = 8

为了让每个字段都能自然对齐，整体基地址 `base` 必须同时满足：

base ≡ 0 (mod 1)  
base ≡ 0 (mod 4)  
base ≡ 0 (mod 8)

换句话说，`base` 必须是所有字段对齐值的**公共倍数**。

因此，复合类型的整体对齐要求应当是：

$$target\_base = align(S) = lcm(align(field_1), align(field_2), ..., align(field_n))$$

也就是所有字段对齐要求的**最小公倍数**(最小公倍数是最宽松的公共倍数要求)。

但在常见 ABI 中，基础类型的对齐值通常是 2 的幂：

$$1, 2, 4, 8, 16, 32, ...$$

而 2 的幂之间有一个重要性质：

对于一组 2 的幂，它们的最小公倍数就是其中的最大值。

例如：

lcm(1, 4, 8) = 8  
lcm(2, 8, 16) = 16  
lcm(4, 16, 32) = 32

所以在普通系统语言中，复合类型整体对齐可以简化为：

$$
align(S) = max(align(field_1), align(field_2), ..., align(field_n))
$$

在各字段对齐要求均等于其长度时(自然对齐)，复合类型整体对齐还可以简化为：

$$
align(S) = max(len(field_1), len(field_2), ..., len(field_n))
$$

复合类型最外层的对齐要求一般由内存分配器(allocator)和操作系统保证。

**这就是“结构体按最大字段对齐”这句存在歧义的八股文的本质来源。它不是随意规定，也不只是经验优化，而是同余约束在 2 的幂对齐体系下的自然结果。这样就一次性满足了复合类型中所有字段对于基地址base在section2中的数学要求(见section2)**

## 4. padding 的作用一：让字段 offset 满足局部约束

上文说过，复合类型对象整体的 `base` 一般由内存分配器(allocator)、栈帧布局或全局变量布局保证；对象内部字段的 `offset` 由编译器负责安排。

例如：

```rust
struct S {  
    a: u8,  
    b: u32,  
}
```

直觉上，如果 `a` 放在 `base` 偏移 0，那么 `b` 似乎可以直接放在 `base` 偏移 1：

offset(a) = 0  
offset(b) = 1

但现在要求 `u32` 在内存中按照 4 字节对齐，即 `b` 的偏移必须满足：

offset(b) ≡ 0 (mod 4)

此时就需要编译器插入 padding：

offset 0: a  
offset 1: padding  
offset 2: padding  
offset 3: padding  
offset 4: b
offset 5: b
offset 6: b
offset 7: b

即：

offset(a) = 0  
offset(b) = 4

这样，只要整个结构体的 `base` 是 4 的倍数，那么：

$$addr(b) = base + 4$$

也一定是 4 的倍数，满足 `u32` 在内存中按照 4 字节对齐的要求(原理见 section2)。

所以 padding 的本质是**要求编译器通过填充修正字段偏移量，从而使每个字段的相对地址满足其同余约束，即满足其对齐要求。**

## 5. padding 的作用二：为什么结构体大小也要补齐？

前面讨论的是**字段在复合类型内部怎么放**，即每个字段的 `offset` 必须满足它自己的对齐要求。

但复合类型还有一个非常重要的使用场景：**数组**。

例如：

```
let arr: [S; 3];
```

数组中的元素必须按固定步长连续排列：

```
base(arr[0]) = base(arr)
base(arr[1]) = base(arr) + size_of::<S>()
base(arr[2]) = base(arr) + 2 * size_of::<S>()
...
base(arr[i]) = base(arr) + i * size_of::<S>()
```

这里的 `size_of::<S>()` 不只是“字段实际占了多少字节”，它还是元素紧密排列的数组中相邻两个 `S` 之间的**步长**。

现在假设复合类型S需满足对齐要求：

```
align(S) = A
```

那么为了让`S`构成的数组`S[]`的每一个元素都满足对齐要求：

```
base(S[i]) ≡ 0 (mod A)
```

如果 `arr[0]` 已经对齐，即：

```
base(arr[0]) ≡ 0 (mod A)
```

那么为了让 `arr[1]` 也对齐，就必须：

```
base(arr[0]) + size_of::<S>() ≡ 0 (mod A)
```

由于 `base(arr[0]) ≡ 0 (mod A)`，所以要求就变成了：

```
size_of::<S>() ≡ 0 (mod A)
```

因此，结构体`S`的类型长度需要向上取整到自身对齐值的整数倍，这就是尾部 padding 的来源。

例如一个结构体所有字段和字段实际只占 28 字节，但整体对齐要求为 8，那么它的大小不能是 28，而要至少补齐到 32，才能保证数组中每一个元素都满足其对齐要求。

至此，我们讨论了如何让复合类型(或复合类型的数组)中所有基本类型的字段都满足其对齐要求。那么为何需要对齐要求？对齐要求的第一性原理是什么？

## 6. 对齐的第一性原理是微架构访存设计，不是为了不跨硬件边界

根据 section2， 一个字段的对齐要求为`align(field)`表示其必须满足：

base_addr(field) ≡ 0 (mod align(field))

很多以偏概全的文章会解释说：

> 对齐是为了避免跨 cache line、内存页等硬件边界影响性能

这个解释只是结果层，不是第一性原理，甚至在逻辑上是不恰当的。"避免跨 cache line、内存页等硬件边界"是对齐带来的有益副产品，但不是它存在的原因。

回到第一性原理，为什么对于cpu直接处理的基本类型会有"对齐"这个要求？

因为在现代微架构中，CPU 执行指令去拿数据的部门叫 Load/Store Unit (LSU)，而它去拿数据的"仓库"是 L1 Data Cache。为了能在一个时钟周期内极速完成读取，L1 Cache 在物理电路上被切分成了若干个固定宽度的访问块（access chunk），LSU 与 Cache 之间的连线就是按这个宽度铺设的。

假设这个访问宽度是 8 字节。Cache 的物理布局就被切成：

```
[0, 8)  [8, 16)  [16, 24)  [24, 32) ...
```

LSU 的一次基本 cache 访问操作对应一个 chunk：指定 chunk 编号，从中取出连续若干字节。访问范围超过一个 chunk，就要么走多次基本操作，要么通过性能影响和要求都更高的复杂电路。

以两个简单的场景为例：

* 场景 A：满足自然对齐的读取（极速通道）
  例如：读取按8字节对齐的地址开始的 u64，由于 addr ≡ 0(mod 8)，u64 的 8 个字节完完整整地躺在 chunk 内。LSU 内部的 MUX 一步选中这个 chunk，数据一次性送回 register，一次基本访问操作搞定。
* 场景 B：不对齐的读取
  例如：读取一个 addr=4 开始的 u64，这个 u64 的 8 个字节的前一半在 chunk [0, 8)，后一半在 chunk [8, 16)。
  对于严格的架构，硬件工程师为了追求极简、高频以及"一次访问就是一次不可分割事务"的原子性，根本没在电路上画"同时从两个 chunk 各抽一半数据并拼装"的复杂交叉电路，所以 LSU直接抛出硬件异常（Alignment Fault），程序被操作系统终止；
  对于宽容的架构(比如x86架构)：硬件会偷偷发两次微指令，读两次拼起来给你，代价是影响性能，且访问的数据越宽，跨 chunk 的代价越高，要求越严格(比如一些SIMD指令仍要求强制对齐)。

所以对齐的本质其实是：用静态地址约束，把微架构访存通路的物理约束一次性消化在编译期，换取跨平台合法性、免费获取的原子性、可控的性能。
